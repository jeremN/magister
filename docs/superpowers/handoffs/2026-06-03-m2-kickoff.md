## 1. What this project is

`concentus-magister` (`/Users/jeremienehlil/Documents/Code/Personal/concentus-magister`) is a from-scratch **Go agent orchestrator**: it routes tasks to AI coding-agent CLIs (Claude/Codex/Gemini) according to declarative DAG *flows*, with approval/verification gates between steps. It's the production-grade successor to a phase-1 prototype called `orch`.

**Locked scope:** single-node, personal, long-running **service** (HTTP/JSON + SSE API, SQLite persistence, resume, concurrent runs), architected so the `Executor` port *can* later generalize into a pluggable workflow engine ‚Äî but that generalization is deferred until the service is proven. v1 targets the full feature set.

## 3. What M2 must deliver

Per spec ¬ß8/¬ß6/¬ß7 and milestone table ¬ß13 (M2 row):

1. **A SQLite implementation of `core.Store`** (the interface already exists in `internal/core/store.go`, unchanged) living in `internal/store/` alongside the existing `Mem`. Methods: `CreateRun`, `SaveStepTransition` (step row + event rows in **one transaction**), `SetRunStatus`, `LoadIncompleteRuns`, `GetRun`, `ListRuns`, `EventsSince`.
2. **Schema + migrations** for `runs`, `steps`, `artifacts`, `events` tables (schema sketched in spec ¬ß8). `events.seq` is an autoincrement that doubles as the SSE replay cursor.
3. **Resume reconciliation** (`LoadIncompleteRuns` + engine logic to continue incomplete runs on startup) per spec ¬ß7 ‚Äî including the documented **at-least-once** caveat for in-flight steps.
4. **persist-then-publish** made real: the synchronous transactional store write happens before the bus publish (the engine already calls `transition()` this way; M2 makes the store side durable).

**Locked technical decisions (from spec ¬ß8 ‚Äî do not re-litigate):**
- Driver: **`modernc.org/sqlite`** (pure-Go, **no cgo**).
- Open with **WAL mode**, `busy_timeout`, `foreign_keys=on`. SQLite has a single writer ‚Üí use a **writer handle with `SetMaxOpenConns(1)`** plus a separate read pool. This is a hard design constraint, not optional.
- Queries via **`sqlc`** (compile-time-checked, parameterized ‚Äî no hand-built SQL strings).
- Migrations embedded via **`//go:embed`**, applied by **`pressly/goose`** on startup.
- `flow_yaml` stored verbatim on the run row = the resume source of truth.

**Fix the three tracked M1 follow-ups during M2** (spec ¬ß17 ‚Äî all are "moot in M1, real under SQLite/concurrent reads"):
- `Store.GetRun` must **deep-copy** the returned `Steps`/`Artifacts` slices (the in-memory `Mem.GetRun` currently aliases internal arrays). The SQLite store returns fresh slices naturally; also fix `Mem` so it's safe for concurrent-read tests.
- `engine.transition` (`internal/engine/engine.go`) must **`return` after publishing the store-error frame** so it does NOT publish the original event when the store write failed.
- `engine.Run` should **not silently discard `SetRunStatus` errors** ‚Äî log them (slog arrives in M3; at minimum surface, per spec ¬ß11 "no silent failures").

**Keep `Mem`** ‚Äî it stays as the fast in-memory store for unit tests.

## 5. Conventions & gotchas to preserve (learned the hard way in M0+M1)

**Process / workflow:**
- This project is built via **brainstorm ‚Üí writing-plans ‚Üí subagent-driven-development**. For each unit: implementer subagent ‚Üí spec-compliance review ‚Üí code-quality review ‚Üí fix loop. Use a capable model (opus) for the riskiest reviews (e.g., the resume/transaction logic).
- Dispatch subagents with the plan **committed in the repo** and point them at the specific task numbers (don't re-paste hundreds of lines). `SendMessage` to continue a subagent is NOT available in this environment ‚Äî dispatch a fresh fix subagent (the code on disk is its context), or apply tiny reviewer-specified fixes directly + verify.
- **Verify reviewer claims yourself** (run the suite, read the diff) ‚Äî reviews caught real bugs, and one implementer's "network unavailable / used cache" still worked.

**Git / commits (user's CLAUDE.md rules ‚Äî strict):**
- **Never** add a `Co-Authored-By` trailer. **Single conventional-commit subject line only**, no body (e.g. `feat(store): add sqlite store`). Detailed rationale goes in docs/PR, not the commit.
- Git identity isn't globally set; commit with `git -c user.name="J√©r√©mie Nehlil" -c user.email="jeremie.nehlil.freelance@proton.me" commit ...`.
- Never use `--no-verify`.

**Environment quirks:**
- An **RTK token-killer hook reformats `go`/`git` command output** (e.g. "Go test: N passed") ‚Äî if output looks summarized, re-run with `-v -count=1` to see raw per-test lines.
- Go module **network fetches work** here (goccy/x-sync downloaded fine). modernc.org/sqlite + goose + sqlc should fetch OK.
- A **UserPromptSubmit hook injects a "secure-by-default libraries" blurb** on every message ‚Äî it's ambient noise, not a user instruction; ignore it (it's JS/Ruby/Python-focused and irrelevant to this Go service). Do keep the spec's actual security stance: trust boundary = loopback, parameterized SQL via sqlc.
- A **Semgrep hook** runs; it flagged `sh -c <command>` in `gate/verifier.go` (CWE-94). That's accepted (operator-supplied flow config, loopback trust boundary) and documented in-code. Expect similar findings; judge against the trust boundary, don't blindly suppress.
- Output style is "learning/explanatory" ‚Äî include brief `‚òÖ Insight` notes when teaching something codebase-specific.

**Code-level gotchas already resolved (keep them this way):**
1. The **engine lives in `internal/engine`, not `internal/core`** ‚Äî putting it in `core` creates an import cycle (`core ‚Üí gate ‚Üí core`). `core` stays a pure domain/ports package.
2. `engine.Run(parent ctx, ...)` derives a child `ctx`; the terminal status switch checks **`parent.Err()` first** (‚Üí canceled) before `firstErr` (‚Üí failed) before default (‚Üí succeeded). Don't reorder.
3. **Deadlock-freedom is structural:** a step waits on dependency channels *before* acquiring any concurrency token, and holds tokens only around the actual work. Preserve this when resume re-wires the DAG.
4. `flow` validation: `Join.OnConflict` is a `FailPolicy` (abort/retry/escalate) ‚Äî the example flow uses `escalate`, and `validateJoin` checks it. `go 1.22` pinned. goccy YAML in **strict mode** (unknown keys error).
5. `Duration` is `flow.Duration` (YAML "5m" string ‚Üí `time.Duration`); convert with `time.Duration(d)`.
