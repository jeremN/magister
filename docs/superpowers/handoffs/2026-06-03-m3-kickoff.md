## 1. What this project is

`concentus-magister` (`/Users/jeremienehlil/Documents/Code/Personal/concentus-magister`) is a from-scratch **Go agent orchestrator**: it routes tasks to AI coding-agent CLIs (Claude/Codex/Gemini) according to declarative DAG *flows*, with approval/verification gates between steps. Single-node, personal, long-running **service** (HTTP/JSON + SSE API, SQLite persistence, resume, concurrent runs). The `Executor` port is the seam that *can* later generalize into a pluggable workflow engine, but that's deferred until the service is proven.

## 3. What M3 must deliver

Per spec §9/§10/§11 and milestone table §13 (M3 row): a running daemon + client, so `cm run --watch` drives `magisterd` end-to-end.

1. **`internal/api`** — stdlib `net/http` (Go 1.22 `ServeMux` method+wildcard patterns, **no router dep**):
   - `POST /v1/runs` (validate flow, generate run ID, `store.CreateRun`, hand to supervisor, return `{id}`)
   - `GET /v1/runs` (list, filter by status) · `GET /v1/runs/{id}` (snapshot: run + steps) · `DELETE /v1/runs/{id}` (cancel)
   - `GET /v1/runs/{id}/events` — **SSE**, with **`Last-Event-ID` replay**: on connect, `store.EventsSince(id, lastSeq)` replays missed rows, then attach to the live bus. (`events.seq` is the cursor; the live frames the engine publishes carry `Seq=0`, so the durable table is the replay source.)
   - `POST /v1/runs/{id}/steps/{step}/approve` — `{approve, reason?}` → resolve a pending gate.
   - `GET /healthz`.
   - Middleware chain: request ID → `slog` logging → panic recovery → auth → timeouts. `http.MaxBytesReader` on bodies; server read/write timeouts; sane security headers.
2. **Supervisor + `ApprovalRegistry`** (spec §5) — the daemon-level owner of active runs and pending gates. This is what makes **manual/escalated gates actually block**: an API-backed `gate.Approver` registers a pending approval keyed by run/step and blocks on a channel until `POST .../approve` resolves it. (Today only `AutoApprover` exists; the `gate.Approver` interface is already the seam.)
3. **`cmd/magisterd`** — config (flags/env/file), wiring (open SQLite store, build engine with executors/gate/join/semaphore/**a real `slog.Logger`**), HTTP server lifecycle, and **graceful shutdown** (SIGTERM cancels all run contexts, flushes, drains the SSE hub within a deadline).
4. **Resume on startup** — wire the M2 pieces: on boot, `store.LoadIncompleteRuns()` → for each, `flow.ParseBytes(run.FlowYAML)` + `Validate` → `engine.Resume(ctx, run, f)`. **`awaiting_gate` resume lands here** (it was deferred from M2): re-emit `gate.awaiting` and re-block on the `ApprovalRegistry`. This means `StepAwaitingGate` becomes a real persisted state the engine sets when a manual/escalated gate blocks — extend the engine's gate path accordingly.
5. **`cmd/cm`** — thin HTTP client, stdlib subcommand dispatch (dependency-light): `run <flow.yaml> [--watch]`, `ls`, `get <run>`, `watch <run>`, `approve|reject <run> <step>`, `cancel <run>`. **Agent-aware:** every command supports `--json` and meaningful exit codes.
6. **`oklog/ulid`** for sortable run IDs (spec §14 — deferred from M2; add it as a pinned, go-1.22-compatible version).
7. **`slog` rollout** (spec §11) — the engine already has a `Log *slog.Logger` field (nil = discard); `magisterd` injects a real handler with `run_id`/`step_id` context. No silent failures.

**Runnable proof (M3 done):** start `magisterd`, `cm run flows/feature-flow.yaml --watch` streams the live journal to completion against the keyless Mock executor; a manual gate blocks and `cm approve` releases it; kill `magisterd` mid-run and restart → it resumes; `go test -race ./...` green including `httptest`-based API tests and the SSE `Last-Event-ID` replay path.

## 5. Conventions & gotchas to preserve

**Process / workflow:**
- brainstorm → writing-plans → subagent-driven-development. Per unit: implementer subagent → spec-compliance review → code-quality review → fix loop. **Verify reviewer claims yourself** (run the suite, read the diff) — the M2 reviews caught real (if minor) issues, and self-reports are optimistic.
- Dispatch subagents pointing at the **committed plan + specific task number** (don't re-paste hundreds of lines); give scene-setting context. `SendMessage` to continue a subagent is **not** available here — dispatch a fresh fix subagent, or apply tiny reviewer-specified fixes directly + verify (then amend the local task commit to keep history clean).
- Use **opus** for the riskiest implement/review (concurrency: SSE hub, approval blocking, shutdown).

**Git / commits (user CLAUDE.md — strict):**
- **Never** a `Co-Authored-By` trailer. **Single conventional-commit subject line only**, no body. Detailed rationale → docs/PR.
- Git identity isn't global: `git -c user.name="Jérémie Nehlil" -c user.email="jeremie.nehlil.freelance@proton.me" commit -m "..."`.
- Never `--no-verify`.

**Environment quirks:**
- An **RTK token-killer hook** reformats `go`/`git` output (e.g. "Go test: N passed"). If output looks summarized, re-run with `-v -count=1` (or `rtk proxy go test ...`) for raw lines.
- Go module **network fetches work** here. `sqlc`/`goose`/`modernc` already cached.
- A **UserPromptSubmit hook** injects a "secure-by-default libraries" blurb on every message — ambient noise (JS/Ruby/Python-focused), ignore it. The actual security stance is the spec's: trust boundary = loopback (`127.0.0.1` default bind), optional static bearer token (constant-time compare) for LAN binds, no cookies → no CSRF, parameterized sqlc SQL.
- A **Semgrep hook** runs; judge findings against the trust boundary (loopback, operator-supplied config), don't blindly suppress. Expect it to look at any new request-handling code.
- Output style is "learning/explanatory" — include brief `★ Insight` notes when teaching something codebase-specific.

**Code-level gotchas to respect (carried + new from M2):**
1. The **engine lives in `internal/engine`, not `internal/core`** (putting it in `core` creates `core → gate → core`). `core` stays a pure domain/ports package.
2. **persist-then-publish:** `engine.transition` writes durably (one tx in SQLite: step + artifacts + events) **then** publishes; on a store-write error it publishes a `store:` error frame and `return`s WITHOUT publishing the original event. The live `event.Event.At`/`Seq` on published frames is best-effort (`Seq=0`); the durable `events` table holds the real autoincrement seq — **the SSE hub must replay from `store.EventsSince`, not from buffered live frames.**
3. **SQLite store design constraint:** one writer handle `SetMaxOpenConns(1)` + a separate reader pool; pragmas (`busy_timeout`, `journal_mode(WAL)`, `foreign_keys(1)`) live in the **DSN** so every connection enforces them. Don't "simplify" this to a single pool. `store.Open(path)` applies goose migrations; it's a startup op (uses goose package globals — not safe to call concurrently).
4. **`core.Store` is frozen** — M2 added SQLite behind the existing interface without changing it. Don't change the interface for M3 either unless the spec demands it; the SSE/snapshot endpoints read through `GetRun`/`ListRuns`/`EventsSince`/`LoadIncompleteRuns` as-is.
5. **`Engine.Resume(parent, rs core.RunState, f)`** exists: it seeds `succeeded` steps and re-runs the rest (at-least-once). M3's startup loop calls it per incomplete run. `Engine.Run` is now a one-liner over a shared `runDAG(parent, runID, f, seed)`.
6. **`Mem` and `SQLite` are equivalent test doubles** for `core.Store` (Mem now implements `LoadIncompleteRuns` too). API/handler unit tests can use `Mem`; integration/resume tests use a real temp SQLite file.
7. **Deadlock-freedom is structural** in the engine (a step waits on dep channels before acquiring any concurrency token). Preserve this if you touch the engine for `awaiting_gate` blocking — the gate block happens *inside* the attempt, after tokens are already held, which is fine, but don't move dep-waits after token acquisition.
8. `flow.Validate` already compiles `conditional` gate exprs and requires arbiter agents for select/synthesize joins — but those gates/joins aren't *evaluated* until M5; M3 manual/auto gates are enough.
