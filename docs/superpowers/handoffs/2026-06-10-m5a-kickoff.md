## What M5a builds

Implement `gate.policy: conditional` — compile `condition.expr` with **`expr-lang/expr`** at submit time and evaluate it synchronously against the gated step's result, replacing today's M1 fallback to manual approval. Guiding model: **a conditional gate is an auto gate whose pass/fail comes from an expr instead of a shell exit code** — so it reuses auto's synchronous + escalate plumbing. The YAML surface already exists (`flow.Condition`, `GateConditional`); M5a only fills in behavior. No store/SSE/schema change.

User-approved design decisions: **expr-lang/expr v1.17.8**; **escalate extends to conditional** (a false condition with `on_fail: escalate` → human approval).

## Gotchas already de-risked (don't rediscover these)

- **Dependency is pre-cached.** `github.com/expr-lang/expr@v1.17.8` is already in the local `GOMODCACHE` (`go env GOMODCACHE`), so Task 0's `go get` and Task 1's `go mod tidy` resolve **offline** — subagents need no network. (If a controller `go get` is ever blocked by the sandbox, it still resolves from cache; disable the sandbox only if truly needed.) The dep declares `go 1.18` and has **zero transitive deps**, so the `go 1.22` ceiling is safe.
- **Empirically verified against the real library** (so the plan's code is correct): `expr:"result"` / `expr:"cost_usd"` struct tags DO produce snake_case env names (`result.cost_usd`, `result.summary contains "OK"` evaluate correctly); `expr.AsBool()` with the typed `expr.Env(GateEnv{})` compile-**rejects** a non-bool expression (e.g. `result.summary`).
- **The mock agent returns `cost_usd 0.01`** (asserted in the existing `TestEscalateApproveSucceeds`), so the integration tests and the manual proof use `result.cost_usd < 1.0` (true) / `> 1.0` (false) — zero cost, no network, no keys.
- **Compile-at-submit** hooks into `flow.Validate` (called at `api/handlers.go:40` and `supervisor.go:136`), so a bad expr fails `POST /v1/runs`, never a running step. The compiled `*vm.Program` is in-memory only; resume recompiles (idempotent).
- **Import direction:** `flow` owns `GateEnv`/`Condition.Compile`/`Eval` (imports `expr`); the `gate` package does the `core.Result → flow.GateEnv` mapping (it already imports both). `flow` must NOT import `core`.
