## What shipped

Distributed tracing for the daemon, **off by default** (no `-otel-endpoint` ⇒ no spans, no socket, byte-for-byte today's runtime). One connected trace per run: `POST /v1/runs` → `run <id>` → `step <id>` → `agent <name>` (+ `gate`/`join`, and `push`/`pr`/`ship` under their delivery requests). Run-scoped logs carry the matching `trace_id`/`span_id`. Spec `docs/superpowers/specs/2026-06-22-otel-tracing-design.md` (READ its "Revision (2026-06-22)" section — authoritative), plan `docs/superpowers/plans/2026-06-22-otel-tracing.md`.

**THE BIG PIVOT (the durable lesson of this slice).** The spec was approved on the premise "OTLP/HTTP exporter, **no grpc**, minimal footprint, go 1.22." During Task 1 that premise proved **false**: the official `otlptracehttp` exporter compiles **59 `google.golang.org/grpc` packages into the binary** — the HTTP and gRPC OTLP exporters share an `internal/otlpconfig` package that imports grpc, so even the HTTP exporter links grpc (`otelx → otlptracehttp → otlptracehttp/internal/otlpconfig → grpc`). It also drags `grpc-gateway`/`genproto`/`proto/otlp`, and `proto/otlp v1.4.0` forces the go directive to **`go 1.22.7`**. `proto/otlp` requires grpc in its own `go.mod` at *every* version, so no version pin escapes it. This was caught by `go list -deps` + `go mod why` (NOT by reading docs), escalated to the user, who chose to **hand-roll the exporter** — exactly the project's Prometheus-`/metrics` move.

**The hand-rolled exporter** (`internal/otelx/otlpjson.go`): a `sdktrace.SpanExporter` that serializes spans to **OTLP-JSON** and POSTs to `{endpoint}/v1/traces` over `net/http`. The OTel **SDK** module does *not* depend on grpc/proto — only the *exporter* modules do — so the dep set is exactly three OTel modules (`otel`, `otel/trace`, `otel/sdk`) at **v1.32.0** (its whole train is bare `go 1.22`; without the exporter module, `proto/otlp`/`auto/sdk` are never pulled). Zero grpc, zero protobuf, zero exporter module, `go 1.22` unchanged. **OTLP-JSON wire-format footguns the code handles** (Opus verified each by dumping a live payload): trace/span IDs as lowercase hex (SDK's `.String()`); 64-bit numerics (timestamps, int attrs) as decimal STRINGS; span kind as a direct int cast (OTel `SpanKind` matches OTLP); and **`status.code` REMAPPED** — OTel SDK `codes` is `Unset=0/Error=1/Ok=2` but OTLP proto is `Unset=0/Ok=1/Error=2`, so a naive cast turns every error span into "OK".

## Key properties (Opus-verified)

- **Off by default ⇒ byte-for-byte unchanged.** No `-otel-endpoint` ⇒ `otelx.Init` returns `(nil,nil)`, global tracer stays no-op, `tp.Shutdown` guarded by `if tp != nil`. The `NewLogHandler` wrap is unconditional but a no-op when no span is active. Pre-existing suites (no provider) stay green — the regression proof.
- **Instrumented packages import only the OTel API** (`otel`/`otel/trace`/`otel/attribute`/`otel/codes`, + `propagation` in api) via a package-level `var tracer`. SDK/exporter live only in `internal/otelx` + `cmd/magisterd`.
- **The async submit→run linkage** (`supervisor.start`): the run goroutine's ctx is `context.WithCancel(context.Background())` (detached, so a run outlives its request) — which severs the trace. Fix: `start(parent, …)` re-seeds the base via `trace.ContextWithRemoteSpanContext(context.Background(), sc)` — carries the TRACE (values), NOT the request's cancellation. Both callers updated (`Submit` passes the request ctx; `ResumeAll` passes `context.Background()`). Run lifetime cannot regress. The run root span ends up a child of the (already-ended) submit span — valid, spans reference parents by ID, not liveness.
- **Config:** `-otel-endpoint` (collector base URL, `/v1/traces` appended if absent) + `-otel-service-name`; `OTEL_EXPORTER_OTLP_ENDPOINT`/`OTEL_SERVICE_NAME` env honored when flags unset (flags win). Telemetry never fatal (Init error → warn + continue). Drain flushes the tracer AFTER `httpSrv.Shutdown` (so delivery spans created during the drain window are captured — the M2 fix).

## Process notes (durable)

- **The grpc-in-the-HTTP-exporter trap is the headline lesson.** "OTLP/HTTP, no grpc" is a *false* assumption for the official OTel exporter — `otlptracehttp` links grpc via the shared `internal/otlpconfig`, and `proto/otlp` requires grpc at every version. Always verify a dependency claim with `go list -deps <pkg> | grep <thing>` and `go mod why -m <module>`, not docs/intuition — this is what caught it. The fix (hand-roll a `SpanExporter` emitting OTLP-JSON over net/http) keeps the project's near-zero-dep invariant and is ~200 lines.
